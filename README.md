# SDD 碎碎念

> 写给自己，也写给每一个想认真用好 SDD 的人。

---

## 你是 PM，你要兜底

SDD 的核心逻辑很简单：**Spec 是一切的起点，代码只是 Spec 的投影。**

但这句话背后藏着一个你必须接受的现实——

> **你写的 Spec，最终你自己来兜底。**

AI 不会替你负责。它只会忠实地执行你写下的每一个字。你写清楚了，它给你一个好产品；你写含糊了，它给你一个"功能上没错、但完全不是你想要的东西"的玩意儿。

所以如果你想用好 SDD，你首先得是一个有功底的 PM。

不是懂写代码，而是懂**把想法精确地变成语言**。需求边界在哪里、用户场景是什么、哪些状态需要处理、哪些是明确不做的——这些你要在 Spec 阶段就想清楚。

**Spec 写得越扎实，后面越省心。Spec 写得越含糊，后面越是一场灾难。**

---

## Spec 是树根，代码是树叶

SDD 的理想状态：

```
Spec → Plan → Tasks → Code
```

每一片叶子都从根长出来。你改了根，叶子跟着变。

实践中分两种情况：

- **小改 Spec**：可以增量同步到代码，AI 看着上下文做局部修改，问题不大。
- **大改 Spec**：不要犹豫，**把所有代码删掉，重新生长一遍。**

是的，这很残忍，也非常烧 Token。但这是正确的做法。

如果你舍不得删，选择在已有代码上打补丁，让 AI 在一个和新 Spec 对不上的旧代码里东修西改——你会得到一坨新旧逻辑混杂、没有人完全理解的东西。代码变屎山只是时间问题。

> 大改就删，这是 SDD 的纪律。破了这条纪律，SDD 就开始腐烂。

---

## SDD 在个人开发者手里最容易退化

这是一个残酷的观察。

SDD 要求你**始终以 Spec 为权威**。每次改动，先改 Spec，再动代码。

但实际上，当你一个人开发、压力大、思路来了、或者 AI 直接帮你改好了代码……你很容易跳过 Spec，直接让 AI 改代码。

改了几次之后，Spec 和代码开始分叉。

又过了几次，你发现 Spec 已经描述的不是当前系统了。

再过一段时间，Spec 就彻底废掉了——没人维护，没人相信，成了一个装饰品。

**于是你回到了 Code-First 的老路。**

只不过这次更糟。因为你还有一份过时的 Spec 在那里，让 AI 在读取上下文时产生混淆。代码和 Spec 一起变成了屎山。

这不是 SDD 的失败，这是**纪律的失败**。

---

## 模型能力是 SDD 的地基

SDD 不是一个宽容的流程。它对 AI 模型的要求很高。

每一步——生成 Spec、生成 Plan、拆解 Tasks、执行实现——每一步都要求 AI 准确理解上下文，做出正确判断。

假设每一步的准确率是 99%：

- 10 步之后：准确率约 90%
- 50 步之后：准确率约 60%
- 100 步之后：准确率约 37%

这就是**比萨斜塔效应**。每一步微小的偏差，经过几十步上百步的累积，最终的结果是一栋歪的楼。

所以 SDD 必须用最强的模型，而且要舍得在每个关键节点**人工介入检查**，而不是无脑全自动流水线跑完。

> 你节省了检查的时间，但会在后期付出十倍的调试成本。

---

## 老项目怎么办：先蒸馏，再重生

如果你有一个老代码库想用 SDD 方式重构，正确的姿势是：

1. **先把老代码精粹成 Spec**——让 AI 通读代码，提炼出它现在"在做什么"，写成一份 Spec
2. **审查并修改这份 Spec**——这份 Spec 可能包含很多你不想要的历史包袱，删掉、改掉、加上新的意图
3. **用新 Spec 生长新代码**——不是在老代码上打补丁，而是重新生成

这个过程痛苦，但值得。你会得到一个和你的意图真正对齐的系统，而不是一个"大概能用"的遗留物。

---

## SDD 不适合小项目

说实话：**SDD 非常烧 Token。**

Constitution → Spec → Clarify → Plan → Tasks → Implement，跑完一遍下来，花费不菲。

对于小型项目，这套流程的 overhead 往往超过了项目本身的复杂度。

**小项目，直接 Freestyle 就好。**

现在的 Claude / Codex 能力已经很强，一个清晰的 prompt 加上几轮对话，完全可以做出一个不错的小工具。你不需要这么重的 harness。

SDD 真正的价值在于：

- **中大型项目**，需求复杂，多人协作，需要对齐
- **长周期项目**，需要在几个月后还能理解系统的每一个决策
- **需求频繁变动**但希望变动有迹可循、可控的项目

如果你只是想快速验证一个想法，别用 SDD，会把自己搞死。

---

## 一句话总结

| 情况 | 建议 |
|---|---|
| 大型项目 / 长周期 / 多人协作 | 用 SDD，认真写 Spec，严守纪律 |
| 小项目 / 快速原型 / 一次性脚本 | 直接 Freestyle，别过度设计 |
| 老项目重构 | 先蒸馏 Spec，再重新生长代码 |
| Spec 小改 | 增量同步到代码 |
| Spec 大改 | 删掉所有代码，从 Spec 重新生长 |
| AI 模型不够强 | 别用 SDD，结果会是比萨斜塔 |

---

> Spec 写得好，代码无所谓——AI 会写。  
> Spec 写得烂，再好的 AI 也救不了你。  
> 纪律破了，SDD 就死了。
