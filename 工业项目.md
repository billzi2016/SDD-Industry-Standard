# 工业项目里的 SDD

---

## 离客户越近，越难

工业项目有一个反直觉的规律：

> **离客户越近的东西，越难做，也越容易变。**

很多人以为最难的部分是后端——高并发、分布式、微服务架构、数据库优化。但这些东西都是**定番**。业界有成熟的解法，文档完善，模式固定，稍微有经验的工程师加上 AI 的辅助，基本上都能搞定。

真正难的，是靠近客户那一侧的东西：

- **前端设计**（不是 UI/UX，而是接口层、交互契约、数据结构的设计）
- **Spec 设计**（需求的边界、优先级、场景覆盖）

这两个东西**无定形**。没有标准答案，没有最佳实践可以照搬，而且每个客户都不一样，每个阶段都在变。

---

## 客户连接的是"根"

在一个工业项目里，系统的结构大致是这样的：

```
客户
  └── 前端设计（接口契约 / API 规范 / 数据模型）
        └── Spec（功能需求 / 边界定义 / 业务逻辑）
              └── 后端实现（高并发 / 数据库 / 服务治理）
                    └── 基础设施（部署 / 监控 / CI/CD）
```

客户接触到的，是最顶层——前端设计，也就是那一套 API 接口、数据契约、交互规范。这就是整棵树的**根**。

根往下，是 Spec。Spec 往下，才是后端实现和基础设施。

后端有多难都无所谓——那是树干和树枝，可以修剪、可以替换。但如果**根是坏的**，整棵树都得推倒重来。

---

## 根坏了，意味着什么

前端设计或 Spec 出了大问题，后果不是局部修复，而是**全面迁移（Migration）**。

- API 结构设计错了 → 所有调用方需要改
- 数据模型定义错了 → 数据库迁移脚本、历史数据处理、所有依赖这个模型的服务全部要动
- Spec 核心逻辑理解偏了 → 从 Plan 到 Tasks 到代码，重来一遍
- 接口契约不稳定 → 前后端、各微服务之间的协作全部进入混乱状态

迁移的成本，不是"改一改"那么简单。它是一种**系统性代价**，而且往往在项目最忙的时候爆发——因为问题都是在集成阶段、上线前才被发现的。

> 根种错了，不是重新修剪枝叶的问题，是要把整棵树挖出来重种。

---

## 为什么前端设计和 Spec 这么难

因为它们面对的是**不确定性**，而不是复杂性。

后端的高并发，是一道复杂的题，但有答案。你可以查文档、看案例、跑 benchmark，最终收敛到一个解。

但前端设计和 Spec 面对的问题是：

- **客户自己也不知道想要什么**，或者说得清楚但想的不清楚
- **需求是动态的**，今天确认的，下周可能推翻
- **边界模糊**，什么在 scope 里、什么不在，永远是争论的来源
- **隐性假设很多**，客户以为你懂的那些"常识"，其实你根本不知道

这就是"无定形"的含义——不是没有规律，而是规律随时在变，而且变化的源头在人，不在技术。

---

## SDD 在工业项目里的意义

正因为如此，SDD 在工业项目里的价值比在小项目里大得多。

SDD 的核心动作是**把所有不确定性前置**——在写代码之前，先把根种好。

`/speckit.constitution` 建立项目原则，是在说：我们做决策的依据是什么。  
`/speckit.specify` 做的是：把客户的模糊意图，转化成精确的功能边界。  
`/speckit.plan` 做的是：把功能边界，映射到具体的接口契约和数据模型。

这个过程不是一次就能做对的。需要 `clarify`，需要反复确认，需要客户参与。但这些摩擦都应该发生在 Spec 阶段——**在代码存在之前**。

> 宁可在 Spec 阶段吵三天，也不要在代码写完之后做 Migration。

---

## 实践中的优先级

工业项目里，精力应该这样分配：

| 层级 | 工作 | 难度 | 优先级 |
|---|---|---|---|
| 前端设计（接口契约） | API 结构、数据模型、交互规范 | 极高 | 最优先，宁慢勿错 |
| Spec 设计 | 功能边界、业务逻辑、边缘场景 | 极高 | 最优先，宁慢勿错 |
| 后端实现 | 高并发、服务治理、数据库 | 中等 | 交给 AI，人工审核关键节点 |
| 基础设施 | 部署、监控、CI/CD | 低 | 模板化处理，基本不需要手动干预 |

AI 能力越强，越往下层的工作越可以放手。但前两层——根——永远需要人来把关。

---

## 一个可以操作的原则

在工业项目里应用 SDD，有一条原则可以作为判断依据：

> **任何一次改动，如果它改变了接口契约或核心数据模型，就必须先改 Spec，评估影响范围，确认迁移成本，再动代码。**

不走这个流程的改动，都是在埋雷。雷埋多了，项目就进入"只有敢动、没人敢删"的状态——每个人都在绕着旧结构修修补补，没有人有勇气也没有人有能力真正重构。

这是工业项目最常见的死法。

---

## 小结

- 离客户近的东西最难：前端设计（接口层）和 Spec
- 后端、高并发这些是定番，AI 搞定，不是瓶颈
- 根是接口契约 + Spec，根坏了 = 全面迁移
- SDD 的价值就是把"根的问题"前置到代码存在之前解决
- 工业项目里，Spec 阶段的每一分钟，都是在省后期迁移的每一个小时
